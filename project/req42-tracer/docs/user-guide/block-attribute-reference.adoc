= AsciiDoc Block Attribute Reference
:toc: left
:toc-title: Contents
:icons: font

req42-tracer reads three types of structured blocks from AsciiDoc files.
Each block is defined by a block attribute line `[type,key=value,...]` followed by a heading `== Title`.

== [req] ‚Äî Requirement

Defines a software or system requirement.

=== Syntax

[source,asciidoc]
----
[req,id=<ID>,version=<N>,priority=<priority>,aspice=<process>,
 status=<status>,reviewed-by=<reviewer>,reviewed-date=<YYYY-MM-DD>,
 derives=<ID1>,<ID2>]
== Requirement Title

Requirement description text.
----

=== Attributes

[cols="1,1,1,4",options="header"]
|===
| Attribute | Required | Example | Description
| `id` | ‚úÖ | `REQ-LOGIN-001` | Unique requirement ID. Convention: `REQ-<AREA>-<NNN>`
| `version` | ‚Äî | `1` | Integer version number (default: 0)
| `priority` | ‚Äî | `high` | `high`, `medium` (default), `low`
| `aspice` | ‚Äî | `SWE.1` | ASPICE process this requirement belongs to
| `status` | ‚Äî | `approved` | `draft` (default), `approved`, `deprecated`
| `reviewed-by` | ‚Äî | `paulefl` | Reviewer identifier
| `reviewed-date` | ‚Äî | `2026-03-15` | Review date in `YYYY-MM-DD` format
| `derives` | ‚Äî | `SYS-001,SYS-002` | IDs of parent requirements (comma-separated)
|===

=== Example

[source,asciidoc]
----
[req,id=REQ-AUTH-001,version=2,priority=high,aspice=SWE.1,
 status=approved,reviewed-by=paulefl,reviewed-date=2026-04-01,
 derives=SYS-003]
== System shall support session-based authentication

The system shall create an authenticated session upon successful login.
Sessions shall expire after 30 minutes of inactivity.

Acceptance Criteria:
- Login with valid credentials creates a session cookie
- Expired sessions redirect to login page
- Session tokens are cryptographically random (256 bit)
----

== [arch] ‚Äî Architecture Element

Defines an architecture component, module, or sub-system.

=== Syntax

[source,asciidoc]
----
[arch,id=<ID>,parent=<parent-ID>,aspice=<process>,
 req=<REQ-ID1>,<REQ-ID2>,impl=<path>,test-spec=<spec-ID>]
== Component Title
----

=== Attributes

[cols="1,1,1,4",options="header"]
|===
| Attribute | Required | Example | Description
| `id` | ‚úÖ | `comp.auth` | Unique element ID. Convention: dot-notation hierarchy `comp.area.sub`
| `parent` | ‚Äî | `comp.system` | Parent element ID. Omit for top-level (Level 1) elements
| `aspice` | ‚Äî | `SWE.2` | ASPICE process level. Auto-derived if `aspice.auto-derive: true`
| `req` | ‚Äî | `REQ-AUTH-001` | Referenced requirement IDs (comma-separated)
| `impl` | ‚Äî | `internal/auth/handler.go` | Implementation file or package path
| `test-spec` | ‚Äî | `TS-AUTH-001` | Linked test specification ID
|===

NOTE: Commas are also attribute separators in AsciiDoc block syntax. To reference multiple requirements, use separate `[arch]` blocks or list them consecutively: `req=REQ-001` (one per block).

=== Hierarchy

[source,asciidoc]
----
[arch,id=comp.system,aspice=SWE.2]           <1>
== System (Level 1 ‚Äî no parent)

[arch,id=comp.auth,parent=comp.system,        <2>
 aspice=SWE.2,req=REQ-AUTH-001,
 impl=internal/auth]
== Authentication Component (Level 2)

[arch,id=comp.auth.session,parent=comp.auth,  <3>
 aspice=SWE.3,req=REQ-AUTH-001,
 impl=internal/auth/session.go]
=== Session Manager (Level 3)
----
<1> Root element (no parent) ‚Äî typically the whole system
<2> Sub-component ‚Äî references requirements via `req=`
<3> Implementation unit ‚Äî `SWE.3` level with `impl=` path

== [dsn] ‚Äî Detailed Design Element

Defines a software detailed design unit (ASPICE SWE.3). Links to an architecture component and is verified by unit tests.

=== Syntax

[source,asciidoc]
----
[dsn,id=<ID>,arch=<ARCH-ID>,aspice=SWE.3,impl=<path>]
== Design Element Title
----

=== Attributes

[cols="1,1,1,4",options="header"]
|===
| Attribute | Required | Example | Description
| `id` | ‚úÖ | `comp.auth.session` | Unique element ID. Typically extends parent arch ID
| `arch` | ‚Äî | `comp.auth` | Parent `[arch]` element ID (SWE.2). Omitting causes an orphan-dsn gap.
| `aspice` | ‚Äî | `SWE.3` | ASPICE process level (default: `SWE.3`)
| `impl` | ‚úÖ | `internal/auth/session.go` | Implementation file path
|===

=== Example

[source,asciidoc]
----
[dsn,id=comp.auth.session,arch=comp.auth,aspice=SWE.3,
 impl=internal/auth/session.go]
== Session Manager

Manages session tokens and expiry logic.
----

== [test-spec] ‚Äî Test Specification

Defines a test specification. Links to requirements (SWE.6), architecture (SWE.5), or design elements (SWE.4).

=== Syntax

[source,asciidoc]
----
[test-spec,id=<ID>,req=<REQ-ID>,arch=<ARCH-ID>,dsn=<DSN-ID>,aspice=<process>]
== Test Specification Title
----

=== Attributes

[cols="1,1,1,4",options="header"]
|===
| Attribute | Required | Example | Description
| `id` | ‚úÖ | `TS-AUTH-001` | Unique spec ID. Convention: `TS-<AREA>-<NNN>`
| `req` | ‚Äî | `REQ-AUTH-001` | Requirement IDs (SWE.6 ‚Äî SW Qualification Test)
| `arch` | ‚Äî | `comp.auth` | Architecture element IDs (SWE.5 ‚Äî Integration Test)
| `dsn` | ‚Äî | `comp.auth.session` | Design element IDs (SWE.4 ‚Äî Unit Test)
| `aspice` | ‚Äî | `SWE.4` | ASPICE process this test covers
|===

NOTE: By convention, use one of `req=`, `arch=`, or `dsn=` to identify the test level. A test-spec with none of these is flagged as orphan by `req42-tracer gaps`. Multiple attributes are accepted by the parser (e.g., a test-spec may link both `req=` and `arch=`).

=== In-code test annotations (convention)

Place `[test-spec]` blocks as comments in your Go test files as a documentation convention.
This keeps the spec and the test code co-located and makes traceability visible in code review.

[source,go]
----
// [test-spec,id=TS-AUTH-001,req="REQ-AUTH-001",aspice="SWE.5.BP3"]
// TestSessionExpiry verifies that sessions expire after 30 minutes of inactivity.
func TestSessionExpiry(t *testing.T) {
    // ...
}
----

NOTE: req42-tracer currently parses only `.adoc` files. The `[test-spec]` comments in Go test files serve as a code convention and are not automatically parsed into the traceability graph. To link tests to requirements, declare `[test-spec]` blocks in your `.adoc` documentation files and reference them via `arch=` or `req=`.

== Traceability Rules (ASPICE PAM 4.0)

req42-tracer enforces the full SWE.1‚ÄìSWE.6 traceability chain:

[source]
----
[req,SWE.1] ‚îÄ‚îÄreq=‚îÄ‚îÄ‚ñ∂ [arch,SWE.2] ‚îÄ‚îÄarch=‚îÄ‚îÄ‚ñ∂ [dsn,SWE.3] ‚îÄ‚îÄimpl=‚îÄ‚îÄ‚ñ∂ src/...
     ‚îÇ                     ‚îÇ                        ‚îÇ
     ‚îî‚îÄ‚îÄreq=‚îÄ‚îÄ‚ñ∂ [test-spec/SWE.6]                  ‚îî‚îÄ‚îÄdsn=‚îÄ‚îÄ‚ñ∂ [test-spec/SWE.4]
                [test-spec/SWE.5] ‚óÄ‚îÄ‚îÄarch=‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îò
----

=== Coverage Rules

[cols="2,1,1,2,3",options="header"]
|===
| Link | Attribute | ASPICE | Required | Example
| `[req]` ‚Üí `[arch]` | `req=` on `[arch]` | SWE.2 BP4 | ‚úÖ | `[arch,req=SWR-001]`
| `[arch]` ‚Üí `[dsn]` | `arch=` on `[dsn]` | SWE.3 BP4 | optional | `[dsn,arch=comp.parser]`
| `[arch]`/`[dsn]` ‚Üí impl | `impl=` | SWE.2/SWE.3 BP5 | ‚úÖ | `impl=src/internal/parser/`
| `[dsn]` ‚Üí Unit Test | `dsn=` + `aspice=SWE.4` | SWE.4 BP4 | ‚úÖ | `[test-spec,dsn=comp.parser.tokenizer,aspice=SWE.4]`
| `[arch]` ‚Üí Integration Test | `arch=` + `aspice=SWE.5` | SWE.5 BP4 | ‚úÖ | `[test-spec,arch=comp.parser,aspice=SWE.5]`
| `[req]` ‚Üí SW Qualification Test | `req=` + `aspice=SWE.6` | SWE.6 BP4 | ‚úÖ | `[test-spec,req=SWR-001,aspice=SWE.6]`
| `[test-spec]` ‚Üí TestResult | JUnit/go-test | SWE.4/5/6 BP5 | ‚úÖ | CI artifact
|===

=== Attribute Placement

[cols="1,2,2",options="header"]
|===
| Attribute | Belongs to | Not to
| `req=` | `[arch]`, `[test-spec/SWE.6]` | ~~`[dsn]`~~
| `arch=` | `[dsn]`, `[test-spec/SWE.5]` | ~~`[req]`~~
| `dsn=` | `[test-spec/SWE.4]` | ~~`[req]`~~, ~~`[arch]`~~
| `impl=` | `[arch]`, `[dsn]` | ~~`[req]`~~
| `aspice=` | all block types | ‚Äî
| `derives=` | `[req]` | ‚Äî
|===

=== Complete Example (full chain)

[source,asciidoc]
----
[req,id=SWR-001,aspice=SWE.1]
== System shall parse AsciiDoc blocks

[arch,id=comp.parser,req=SWR-001,aspice=SWE.2,impl=src/internal/parser/]
== Parser Component

[dsn,id=comp.parser.tokenizer,arch=comp.parser,aspice=SWE.3,impl=src/internal/parser/tokenizer.go]
== Tokenizer Unit

[test-spec,id=TS-UNIT-001,dsn=comp.parser.tokenizer,aspice=SWE.4]
== Unit Test: Tokenizer
run: go test ./src/internal/parser/... -run TestTokenize

[test-spec,id=TS-INT-001,arch=comp.parser,aspice=SWE.5]
== Integration Test: Parser + Graph
run: go test ./src/internal/... -run TestParserGraphIntegration

[test-spec,id=TS-SWR-001,req=SWR-001,aspice=SWE.6]
== SW Qualification Test: Parser extracts all req blocks
run: go test ./src/internal/parser/... -run TestParseAllFromDir
----

=== Gap Report Messages

| Message | Meaning | Fix |
|---|---|---|
| `orphan requirement` | `[req]` has no `[arch]` with `req=` | Add `[arch,req=SWR-XXX]` |
| `missing impl` | `[arch]`/`[dsn]` has no `impl=` | Add `impl=src/...` |
| `untested architecture element (SWE.5)` | `[arch]` has no `[test-spec]` with `arch=` | Add Integration Test |
| `orphan design element (SWE.3)` | `[dsn]` has no `arch=` parent | Add `arch=` to `[dsn]` |
| `untested detailed design (SWE.4)` | `[dsn]` has no `[test-spec]` with `dsn=` | Add Unit Test |
| `untraced test result` | `[test-spec]` has no matching TestResult in JUnit/go-test | Check CI output |

`req42-tracer trace` traverses this chain and reports coverage at each level.

== Naming conventions

[cols="1,2",options="header"]
|===
| Artifact | Convention
| Requirements | `REQ-<AREA>-<NNN>` e.g. `REQ-AUTH-001`, `REQ-PERF-002`
| System-level | `SYS-<NNN>` e.g. `SYS-001`
| Architecture | dot-notation `comp.<area>.<sub>` e.g. `comp.auth.session`
| Test specs | `TS-<AREA>-<NNN>` e.g. `TS-AUTH-001`
|===
