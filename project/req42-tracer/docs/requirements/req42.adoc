= req42-tracer — Requirements (REQ42)

:description: req42-tracer Requirements Tracing & ASPICE Compliance Tool
:module: github.com/paulefl/req42-tracer

== Overview

This document defines the functional and non-functional requirements for req42-tracer using AsciiDoc [req] blocks with structured metadata.
Each requirement is tagged with a unique ID, version, priority, status, and ASPICE classification.

== Functional Requirements

=== CLI & Project Initialization

[req,id=REQ-INIT-001,version=1,priority=high,aspice=SWE.1,
 status=approved,reviewed-by=paulefl,reviewed-date=2026-05-07]
== System shall initialize new REQ42+ARC42 projects

[test-spec,id=TS-INIT-001,req=REQ-INIT-001,aspice=SWE.4-BP3]
====
Test: req42-tracer init creates project structure with all template files
====

The system shall provide `req42-tracer init` command that creates a new project with:
- Interactive prompts for project name, module path, description, and optional directory
- Automated mode (--interactive=false) with explicit flags for CI/CD automation
- Pre-configured project templates: req42.adoc, arc42.adoc, architecture.jsonc, .req42.yaml, .gitignore
- Directory creation if not present (--dir flag support)

Acceptance Criteria:
- `req42-tracer init` in interactive mode prompts for all project details
- `req42-tracer init --dir=./my-project --name=MyProject --module=github.com/user/myproject --interactive=false` creates project non-interactively
- All template files are correctly substituted with project metadata
- docs/requirements, docs/arc42, reports directories are created

=== AsciiDoc Parser

[req,id=REQ-PARSE-001,version=1,priority=high,aspice=SWE.1,
 status=approved,reviewed-by=paulefl,reviewed-date=2026-05-07]
== System shall parse AsciiDoc [req] blocks with metadata

[test-spec,id=TS-PARSE-001,req=REQ-PARSE-001,aspice=SWE.4-BP3]
====
Test: ParseAllFromDir extracts all [req] blocks with correct metadata from AsciiDoc
====

The system shall extract all [req,...] blocks from AsciiDoc files with the following attributes:
- id (required): Unique requirement identifier (e.g., SWE-001)
- version (optional): Integer version number
- priority (optional): high, medium, low
- status (optional): draft, approved, deprecated
- aspice (optional): ASPICE process ID (SWE.1, SWE.2, etc.)
- reviewed-by (optional): Reviewer name
- reviewed-date (optional): Date in YYYY-MM-DD format
- derives (optional): Comma-separated list of parent requirement IDs

The parser shall handle multi-line block attribute syntax and extract the first == heading as the requirement title.

Acceptance Criteria:
- All example requirements from req42.adoc are parsed correctly
- Multi-line block attributes (spanning lines) are properly concatenated
- All optional attributes are preserved in the Requirement struct
- Missing 'id' attribute causes the requirement to be skipped with an error log

[req,id=REQ-PARSE-002,version=1,priority=high,aspice=SWE.1,
 status=approved,reviewed-by=paulefl,reviewed-date=2026-05-07]
== System shall parse AsciiDoc [arch] blocks with hierarchy support

[test-spec,id=TS-PARSE-002,req=REQ-PARSE-002,aspice=SWE.4-BP3]
====
Test: ParseAllFromDir extracts all [arch] blocks with parent-child hierarchy
====

The system shall extract all [arch,...] blocks from AsciiDoc files with the following attributes:
- id (required): Unique architecture element identifier (e.g., comp.api)
- parent (optional): Parent element ID for hierarchical decomposition (indicates ARC42 level 2+)
- req (optional): Comma-separated list of requirement IDs this element satisfies
- aspice (optional): ASPICE process ID (auto-derived from hierarchy if not present)
- impl (optional): Implementation pointer (file path or code reference)
- test-spec (optional): Linked test specification ID

The parser shall derive ASPICE levels automatically: level 1 (no parent) → SWE.2; level 2+ (with parent) → SWE.3.

Acceptance Criteria:
- All [arch] blocks are extracted with correct attribute parsing
- Parent-child hierarchies are preserved
- ASPICE levels are auto-derived when not manually specified
- Architecture elements can reference requirements via 'req' attribute

[req,id=REQ-PARSE-003,version=1,priority=high,aspice=SWE.1,
 status=approved,reviewed-by=paulefl,reviewed-date=2026-05-07]
== System shall load Bausteinsicht architecture models with JSONC

[test-spec,id=TS-PARSE-003,req=REQ-PARSE-003,aspice=SWE.4-BP3]
====
Test: BausteinsichtParser loads architecture.jsonc and flattens to dot-notation IDs
====

The system shall load and parse Bausteinsicht JSONC models (architecture.jsonc) with:
- Nested element hierarchy (arbitrary depth)
- Flattening to dot-notation IDs (e.g., system.backend.api)

The loaded model is merged into the traceability graph and can be cross-referenced by requirement IDs.

Acceptance Criteria:
- architecture.jsonc is parsed without errors
- Nested hierarchies are flattened to dot-notation paths
- JSONC comments are properly stripped
- Architecture elements are added to the graph as ArchElements

=== Traceability Graph & Analysis

[req,id=REQ-GRAPH-001,version=1,priority=high,aspice=SWE.2,
 status=approved,reviewed-by=paulefl,reviewed-date=2026-05-07]
== System shall construct and analyze traceability graph

[test-spec,id=TS-GRAPH-001,req=REQ-GRAPH-001,aspice=SWE.5-BP4]
====
Test: Builder constructs bidirectional traceability graph from requirements and architecture
====

The system shall build a traceability graph with:
- Nodes: Requirements, Architecture elements, Test specifications, Test results
- Edges: satisfies, implements, verifies, derives
- Link resolution: explicit annotations (primary) + name-based matching (fallback)
- Bidirectional trace: requirement → architecture → test → result

The graph builder shall merge parsed artifacts (AsciiDoc + Bausteinsicht) and detect/report circular dependencies and orphan artifacts.

Acceptance Criteria:
- Traceability graph is constructed from parsed requirements, architecture, and test specs
- Cross-references are resolved bidirectionally
- Orphan detection identifies unreferenced elements
- Circular dependency detection prevents invalid links

=== ASPICE Compliance Checking

[req,id=REQ-ASPICE-001,version=1,priority=high,aspice=SWE.2,
 status=approved,reviewed-by=paulefl,reviewed-date=2026-05-07]
== System shall validate ASPICE PAM 4.0 compliance

[test-spec,id=TS-ASPICE-001,req=REQ-ASPICE-001,aspice=SWE.5-BP4]
====
Test: Checker.CheckCompliance calculates correct coverage percentages per ASPICE process
====

The system shall provide `req42-tracer aspice` command that:
- Validates configured ASPICE processes (SWE.1–SWE.6, SYS.1–SYS.3)
- Checks best practices: traceability, testability, consistency, bidirectionality
- Calculates coverage percentages per process
- Reports missing / partially satisfied best practices
- Derives ASPICE levels automatically from ARC42 hierarchy (level 1 → SWE.2, level 2+ → SWE.3)

Acceptance Criteria:
- `req42-tracer aspice` outputs coverage report for all configured processes
- BP4 (Traceability) check ensures all requirements have architecture mappings
- BP6 (Testability) check ensures all requirements have test specifications
- Coverage percentages are calculated correctly (N satisfied / N total)

=== Report Generation

[req,id=REQ-REPORT-001,version=1,priority=high,aspice=SWE.2,
 status=approved,reviewed-by=paulefl,reviewed-date=2026-05-07]
== System shall generate traceability reports in multiple formats

[test-spec,id=TS-REPORT-001,req=REQ-REPORT-001,aspice=SWE.5-BP4]
====
Test: TableReporter outputs valid text/markdown/JSON formats with all trace data
====

The system shall provide `req42-tracer trace` command with output formats:
- **text**: ASCII table with requirement→architecture→test columns
- **markdown**: Markdown table suitable for README/documentation
- **json**: Machine-readable JSON with full graph structure

All formats shall support filtering by project, ASPICE process, status, and version.

Acceptance Criteria:
- `req42-tracer trace --format=markdown` produces valid Markdown
- JSON format includes all nodes, edges, and metadata
- Text format is human-readable with aligned columns
- Gap analysis (orphans, missing implementations) is included in text output

[req,id=REQ-REPORT-002,version=1,priority=high,aspice=SWE.2,
 status=approved,reviewed-by=paulefl,reviewed-date=2026-05-07]
== System shall provide gap analysis reporting

[test-spec,id=TS-REPORT-002,req=REQ-REPORT-002,aspice=SWE.5-BP4]
====
Test: Analyzer.AnalyzeGaps identifies orphan requirements and missing implementations
====

The system shall provide `req42-tracer gaps` command that identifies:
- **orphans**: Architecture elements without requirement mappings
- **missing**: Requirements without architecture implementations
- **stale**: References to outdated requirement versions
- **untested**: Requirements without test specifications

Gap report shall show file path, line number, and remediation guidance.

Acceptance Criteria:
- Gap analysis identifies all missing trace links
- Each gap includes source location (file:line)
- Report prioritizes by severity (error/warning/info)
