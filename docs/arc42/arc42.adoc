== 4. Solution Strategy

=== 4.1 Overview

The tool follows a graph-based architecture:

1. Parse AsciiDoc blocks into structured model
2. Load Bausteinsicht architecture model
3. Build traceability graph (requirements → architecture → tests)
4. Analyze for gaps, coverage, ASPICE compliance
5. Generate reports (CLI, HTML)

[arch,id=strategy.graph,req=REQ-GRAPH-001,REQ-VALIDATE-001]
==== Traceability Graph

- Nodes: Requirement, Architecture Element, Test Spec, Test Code, Test Result
- Edges: satisfies, implements, verifies, derives
- Analysis: coverage, orphans, stale-detection

== 6. Runtime View

Describes dynamic behavior during execution.

[arch,id=runtime.trace,aspice=SWE.2,req=REQ-REPORT-001,REQ-VALIDATE-001]
==== Trace Command Sequence

1. Load configuration (.req42.yaml)
2. Parse requirements (AsciiDoc)
3. Load architecture model (Bausteinsicht)
4. Parse test specifications
5. Load test results (JUnit XML / go-test JSON)
6. Build traceability graph
7. Generate report

[arch,id=runtime.watch,aspice=SWE.2,req=REQ-WATCH-001]
==== Watch Mode Sequence

1. Start file watcher (fsnotify)
2. On file change: rebuild graph, regenerate HTML
3. Trigger live-reload in browser

== 12. Glossary

[glossary]
Requirement::
  A functional or non-functional system requirement, tagged with `[req,...]` in AsciiDoc.

Architecture Element::
  A system component or sub-component, tagged with `[arch,...]` in AsciiDoc.

Test Specification::
  A specification of test scenarios, tagged with `[test-spec,...]` in AsciiDoc.

Test Result::
  The outcome of a test execution, parsed from JUnit XML or go-test JSON.

Traceability::
  The bidirectional links between requirements, architecture, and tests.

ASPICE::
  Automotive SPICE (Software Process Improvement and Capability dEtermination), version PAM 4.0.

Graph::
  The internal dependency graph connecting all traced artifacts.
