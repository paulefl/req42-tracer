= ADR-003: Hybrid Test-Result Matching Strategy

**Status:** Accepted
**Decision Date:** 2026-05-07
**Last Updated:** 2026-05-07

== Problem Statement

Test results come from CI/CD pipelines as JUnit XML or go-test JSON artifacts:
- Test names: `TestAuthLogin`, `test_api_auth_endpoint`
- Test status: pass, fail, skip
- Test metadata: duration, error message, stack trace

These results must be linked to requirements and architecture for ASPICE compliance verification (SWE.5: "Verifying the Software Code against the Software Requirements Specification").

The question: **How do we match test results to requirements and test specifications when the CI artifacts don't include explicit requirement IDs?**

== Context & Constraints

1. **Test naming conventions vary:**
   - Go tests: `func TestAuthLogin()` → function name is test identifier
   - JUnit tests: `<testcase name="api.auth.endpoint" classname="ApiTests">` → QName varies
   - Python tests: `def test_api_auth()` → function name
   - Other frameworks: no standard

2. **Requirements and test-specs have explicit IDs:**
   - Requirements: SWE-001, SWE-002 (from [req] blocks)
   - Test specs: spec.api.auth (from [test-spec] blocks)
   - Architecture: comp.api (from [arch] blocks)

3. **Matching information sources:**
   - **Explicit:** [test-code] blocks can map `file=tests/api_test.go, fn=TestAuthLogin` → `spec=spec.api.auth` → `req=SWE-001`
   - **Implicit:** Test name `test_api_auth` might match test-spec ID `spec.api.auth` by convention
   - **Metadata:** JUnit classname, package name could hint at architecture (comp.api)

4. **ASPICE requirement:**
   - SWE.5 BP3: Artifacts must show "which code unit tested which requirement"
   - Manual annotations (reviewed-by) needed for compliance, not algorithmic matching

== Decision: Three-Tier Hybrid Matching

**Explicit links (tier 1) take precedence; name-based fallback (tier 2) with warnings; custom metadata (tier 3) for advanced cases.**

### Tier 1: Explicit Annotations (Primary)

**[test-code] blocks in AsciiDoc** define test-to-requirement mappings with full lineage.

```asciidoc
[test-code,id=test.api.auth,spec=spec.api.auth,
 file=tests/test_api_auth.go,fn=TestAuthEndpoint,
 reviewed-by=paulefl,reviewed-date=2026-05-07]
== API Authentication Test Code

Implements spec.api.auth which verifies SWE-001.
```

**Advantages:**
- Explicit, auditable, reviewable (git commit history)
- Can be signed by reviewer (reviewed-by, reviewed-date)
- ASPICE-compliant (manual verification)
- Handles complex test-to-requirement mappings (N:M)

**Limitations:**
- Requires manual creation and maintenance
- Not scalable for 1000+ tests

### Tier 2: Name-Based Fallback (Secondary)

**Inferred matching from test names to test-spec IDs.**

Algorithm:
1. Extract test identifier from JUnit/go-test artifact:
   - Go: `TestAuthLogin` → `auth-login` (convert CamelCase to kebab-case)
   - JUnit classname: `ApiAuthTest` + testname `endpoint` → `api-auth-endpoint`
2. Normalize test-spec ID: `spec.api.auth` → `api-auth`
3. Similarity match: Levenshtein distance or substring match
4. Generate warning link with confidence score (0-100%)

**Advantages:**
- No manual annotation needed for common cases
- Scales to large test suites
- Catches obvious mappings automatically

**Limitations:**
- False positives: `TestAuth` could match both `spec.api.auth` and `spec.oauth.auth`
- False negatives: Test named `test_endpoint_201` won't match `spec.api.auth`
- Needs explicit review and override

### Tier 3: Custom Metadata (Advanced)

**Optional metadata in JUnit XML or go-test JSON for explicit test-to-requirement mapping.**

Example (JUnit properties):
```xml
<testcase name="TestAuthEndpoint" classname="ApiTests">
  <properties>
    <property name="req42:spec" value="spec.api.auth" />
    <property name="req42:requirement" value="SWE-001" />
    <property name="req42:arch" value="comp.api" />
  </properties>
  <system-out>...</system-out>
</testcase>
```

**Advantages:**
- Explicit mapping embedded in test artifact
- No additional .adoc files to maintain
- CI/CD friendly (generated by test runner)

**Limitations:**
- Requires test framework integration (Go, Python, JUnit plugins)
- Not standard across all frameworks
- Adds CI/CD complexity

== Implementation

### Phase 1: Explicit + Fallback (MVP)

1. **Explicit tier (Tier 1):**
   - internal/parser/adoc.go scans [test-code] blocks
   - Stores TestCode with full lineage (id, spec, file, fn, reviewed-by, reviewed-date)
   - graph/build.go creates TraceLinks from TestCode → TestSpec → Requirement

2. **Fallback tier (Tier 2):**
   - internal/testresult/loader.go parses JUnit/go-test artifacts
   - For each test, normalize name to kebab-case
   - Try to match against test-spec IDs using fuzzy string matching
   - Generate `[test-result]` artifact with confidence score
   - Link: TestResult → TestSpec (if confidence > 80%) with warning if <90%

3. **Warnings:**
   - `req42-tracer trace` and `req42-tracer gaps` flag low-confidence matches
   - Example: `⚠ Test 'test_endpoint' tentatively links to 'spec.api.auth' (confidence: 75%)`

### Phase 2: Custom Metadata (Post-MVP)

- Add tooling to extract req42 properties from JUnit XML / go-test JSON
- Document property schema in CONTRIBUTING.md
- Validate metadata during `req42-tracer validate`

## Consequences

=== Positive

- **Flexibility:** Teams choose annotation style (explicit, fallback, or metadata)
- **Scalability:** Fallback tier handles common case without manual work
- **ASPICE compliance:** Explicit tier provides auditability; confidence scores enable risk assessment
- **Incremental adoption:** Start with fallback, add explicit annotations for critical paths
- **Extensible:** Phase 2 can add metadata tier without changing core logic

=== Negative

- **Complexity:** Three matching strategies to test, debug, document
- **False positives:** Fallback tier can create misleading trace links
- **Maintenance burden:** Explicit annotations must be kept in sync with test code
- **Performance:** Fuzzy string matching scales O(n*m) with test count × spec count

=== Mitigations

- Confidence score filtering: only links >threshold appear in reports; others as warnings
- Clear documentation: distinguish explicit vs. inferred in output
- Dry-run mode: `req42-tracer trace --dry-run` shows proposed matches without committing
- Validation checklist: CI job flags low-confidence matches requiring review

== Alternatives Considered

=== Alternative 1: Explicit Annotations Only

**Pros:**
- ASPICE-compliant by design
- No false positives
- Reviewers control every link

**Cons:**
- Manual work for 1000+ tests
- High barrier to entry for small projects
- Maintenance burden (keep in sync with test code)

**Verdict:** Too strict for MVP; fallback tier enables easier onboarding.

=== Alternative 2: Name-Based Matching Only

**Pros:**
- Zero manual annotation
- Fully automated
- Low friction

**Cons:**
- High false positive/negative rate
- Not ASPICE-compliant without manual review
- Gives false confidence in coverage

**Verdict:** Not compliant; explicit tier needed for SWE.5 BP3.

=== Alternative 3: Test Naming Convention Enforced

**Pros:**
- Consistent mapping: `spec.api.auth` → `test_api_auth`
- Compile-time checking (test names matched at build time)

**Cons:**
- Breaking change for existing test code
- Brittle (renaming tests breaks trace)
- Can't handle complex N:M mappings

**Verdict:** Too rigid; hybrid approach more flexible.

=== Alternative 4: Manual Matrix in .req42.yaml

**Pros:**
- Single source of truth for all test mappings
- ASPICE-compliant (reviewed manually)

**Cons:**
- Centralized file becomes bottleneck for large teams
- Hard to keep in sync (test added, test removed)
- Not versionable per-test (one big matrix diff)

**Verdict:** ADR-003 hybrid is more maintainable; explicit [test-code] blocks are per-test.

**Phase 2 (HTML & Watch):**
- Custom metadata: support in JUnit/go-test parsers
- UI: visualize confidence levels in report

**Phase 3 (IDE Integration):**
- LSP diagnostics: warn about unmapped tests
- Auto-annotation: suggest [test-code] blocks based on fallback matches

**Phase 5 (Production Ready):**
- Custom rules: orgs define their own matching strategies
- Performance: optimize fuzzy matching with caching
