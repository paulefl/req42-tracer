= ADR-001: AsciiDoc Blocks as Primary Source of Truth

**Status:** Accepted
**Decision Date:** 2026-05-07
**Last Updated:** 2026-05-07

== Problem Statement

req42-tracer must manage requirements (REQ42), architecture (ARC42), and test specifications across multiple formats:
- AsciiDoc documents (.adoc files) — human-readable, version-control friendly
- Bausteinsicht JSONC models (architecture.jsonc) — structured, IDE-friendly
- Test result artifacts (JUnit XML, go-test JSON) — CI/CD generated

The question: **Which format is the primary source of truth for requirements and architecture traceability?**

== Context & Constraints

1. **Requirements come from different sources:**
   - Manual [req] blocks in AsciiDoc (REQ42)
   - Manual [arch] blocks in AsciiDoc (ARC42)
   - Bausteinsicht JSONC models (nested hierarchies)
   - Test results from CI/CD pipelines (dynamic, generated)

2. **ASPICE compliance requirements:**
   - Bidirectional traceability must be visible
   - Architectural hierarchy (Whitebox Level 1 → SWE.2, Level 2+ → SWE.3) must be preserved
   - Changes must be auditable and reviewable

3. **Tool objectives:**
   - Learnability: new users productive within 30 minutes
   - IDE support: JSON Schema validation for JSONC
   - LLM-friendliness: AI agents can read/write requirements

4. **Developer workflow:**
   - Architects write [arch] blocks in ARC42 doc
   - Requirements engineers write [req] blocks in REQ42 doc
   - Cross-links (req=, arch=) are explicit in block attributes
   - Bausteinsicht model provides visual hierarchy, backsynced from .adoc

== Decision: AsciiDoc Blocks as Primary Source

**AsciiDoc [req], [arch], [test-spec] blocks are the primary source of truth.**

### Rationale

| Dimension | Preference | Why |
|-----------|------------|-----|
| **Auditability** | AsciiDoc wins | Git diffs show exact attribute changes; reviewers see rationale |
| **Editability** | AsciiDoc wins | Humans write [req] blocks directly; no code generation needed |
| **Hierarchy preservation** | Tie | AsciiDoc [arch] parent= captures level; JSONC nesting also works |
| **IDE support** | JSONC wins | JSON Schema provides autocompletion; .adoc is just text |
| **Bidirectional sync** | AsciiDoc wins | AsciiDoc is versionable; Bausteinsicht model can be auto-generated |
| **ASPICE compliance** | AsciiDoc wins | Manual annotations explicit; reviewers sign off on blocks |

### Implementation

1. **AsciiDoc [req] blocks** define requirements
   - Attributes: id, version, priority, status, aspice, reviewed-by, reviewed-date, derives
   - Title: first == heading after block
   - Description: paragraph(s) following title
   - Parser: internal/parser/adoc.go scans .adoc files for [req,...] blocks

2. **AsciiDoc [arch] blocks** define architecture elements
   - Attributes: id, parent, req, aspice, impl, test-spec
   - Hierarchy: parent= indicates nesting level (for ASPICE derivation)
   - Title: first == heading after block
   - Parser: internal/parser/adoc.go scans .adoc files for [arch,...] blocks

3. **Bausteinsicht JSONC model** is secondary
   - Used for visual representation and rich hierarchy
   - Architecture elements from JSONC are imported into the graph
   - If conflicts arise (same ID in both), AsciiDoc [arch] block wins
   - JSONC can be backsynced from AsciiDoc via a future tool

4. **Test specifications** can be in AsciiDoc or embedded in test code
   - Primary path: [test-spec] blocks in AsciiDoc
   - Secondary path: test-code annotations in source files
   - Test results: loaded dynamically from CI artifacts (JUnit, go-test JSON)

== Consequences

=== Positive

- **Auditability:** Every requirement change is a git commit with diff
- **Reviews:** Reviewers see full context (why, who, when) in AsciiDoc
- **ASPICE compliance:** Manual review signatures (reviewed-by, reviewed-date) are explicit
- **Simplicity:** Single source of truth reduces sync logic
- **Offline-friendly:** Users can edit .adoc files without tooling

=== Negative

- **IDE friction:** .adoc files need syntax highlighting plugin
- **Validation friction:** Schema validation must happen at parse time, not edit time
- **Bausteinsicht sync:** Visual hierarchy must be kept in sync with AsciiDoc parent= attribute
- **Scalability:** Very large documents (1000+ requirements) may become unwieldy

=== Mitigations

- Provide LSP (Language Server Protocol) to enable autocomplete/diagnostics in editors
- Keep REQ42 and ARC42 documents focused (split across chapters as needed)
- Document best practices for using parent= hierarchy correctly
- Automate sync checks: CI job ensures AsciiDoc [arch] matches Bausteinsicht structure

== Alternatives Considered

=== Alternative 1: Bausteinsicht JSONC as Primary Source

**Pros:**
- JSON Schema enables IDE autocompletion
- Structured, machine-processable
- Easier to validate at parse time

**Cons:**
- Not human-readable for non-technical stakeholders
- Harder to review in pull requests (deep nested structures)
- Audit trail is unclear (who decided on parent? when?)
- ASPICE compliance requires explicit signatures, which JSONC doesn't support well

**Verdict:** Rejected — ASPICE PAM 4.0 requires manual review signatures.

=== Alternative 2: Dual Source (AsciiDoc + JSONC as Co-Equals)

**Pros:**
- Flexibility: teams can choose their preferred format
- IDE support + auditability
- Bausteinsicht becomes first-class requirement source

**Cons:**
- Sync complexity: which wins on conflict?
- Unclear to users which source is canonical
- Maintenance burden: keep two formats in sync

**Verdict:** Rejected — single source principle is simpler.
